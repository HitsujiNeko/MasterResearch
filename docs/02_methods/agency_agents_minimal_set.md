# 研究専用エージェント最小セット運用ガイド

**最終更新**: 2026-03-17  
**関連ドキュメント**: [analysis_workflow.md](analysis_workflow.md), [research_guide.md](../01_planning/research_guide.md), [CodingRule.md](CodingRule.md)  
**前提知識**: RQ1-RQ3、LST算出フロー、都市構造パラメータ定義

---

## 1. 目的

本ドキュメントは、`agency-agents` のうち本研究で実用性が高いエージェントのみを選抜し、
LST・都市構造・空間スケール依存の固有文脈を反映した最小運用を定義する。

対象は以下の4点である。

1. 選抜エージェント一覧
2. 固有文脈テンプレート
3. 工程別の呼び出し順
4. 出力品質チェック表

---

## 2. 選抜エージェント一覧（最小6体）

| 区分 | エージェント | 主担当 | 主な使用タイミング |
|------|--------------|--------|--------------------|
| データ整備 | Data Engineer | 入力データ整合、前処理フロー設計 | 前処理開始時、データ更新時 |
| 実装品質 | Code Reviewer | Python/GISコードの欠陥検出 | 実装後、PR前 |
| 文書化 | Technical Writer | 手法・結果の記述標準化 | 図表作成後、章執筆時 |
| 品質ゲート | Reality Checker | 根拠不足・再現性不足の検出 | 提出前、章確定前 |
| 空間妥当性 | Academic Geographer | 空間解釈・地理的一貫性検証 | RQ2/RQ3考察時 |
| 統合管理 | Product Manager | タスク分解・優先順位管理 | 週次計画、進捗再計画時 |

---

## 3. 共通固定コンテキスト（全エージェントに毎回付与）

以下を、各エージェントへの指示の先頭に貼り付ける。

```text
【研究固定コンテキスト】
- 研究テーマ: ベトナム主要都市のLSTと都市構造の関係性評価
- RQ1: どの説明変数がLSTに支配的か
- RQ2: 空間集計単位・解析スケールの違いで関係がどう変わるか
- RQ3: データ制約下でどの程度説明可能か
- LSTは必ず摂氏(℃)で扱う
- 座標系は原則WGS84(EPSG:4326)、面積・距離計算時は投影座標系を使用
- 分析対象域は「ROI全体」と「測量データ有効域」を区別して扱う
- 近傍効果は最低でも 0m, 30-60m, 60-90m, 90-120m の比較を含める
- 結果は再現可能性を担保（入力、処理条件、出力先、乱数シードを明記）
- 推測ではなく、確認可能な根拠に基づいて出力する
```

## 4. エージェント別の追加コンテキスト

### 4.1 Data Engineer

```text
【追加指示: Data Engineer】
- data/input, data/output, data/csv/analysis のデータフローを明示する
- 欠損、CRS不整合、ジオメトリ異常の検出手順を先に示す
- 出力は「処理手順」「失敗時の分岐」「中間保存先」を必ず含める
```

### 4.2 Code Reviewer

```text
【追加指示: Code Reviewer】
- PEP 8、日本語docstring、1関数1責務、pathlib.Path利用を必須基準としてレビュー
- 指摘は「重大度」「根拠」「修正案」「再発防止」を1セットで示す
- LST単位(℃)とCRS処理の誤りを最優先で検出する
```

### 4.3 Technical Writer

```text
【追加指示: Technical Writer】
- 論文向けに「目的→方法→結果→限界→次ステップ」の順で整理
- RQ1-RQ3への対応を段落ごとに明示する
- 手法説明ではデータ制約と仮定を必ず明記する
```

### 4.4 Reality Checker

```text
【追加指示: Reality Checker】
- 根拠のない主張、再現不能な手順、未検証の図表を必ず指摘
- 「この結果で査読を通せるか」の観点で不足を列挙する
- 検証漏れがある場合は、追加実験の最小セットを提示する
```

### 4.5 Academic Geographer

```text
【追加指示: Academic Geographer】
- 空間スケール依存の解釈を優先し、単純相関だけで結論を出さない
- 都市中心部と周縁部のバイアス可能性を点検する
- 土地利用・水域・道路ネットワークの空間配置から説明を補強する
```

### 4.6 Product Manager

```text
【追加指示: Product Manager】
- 週次で「必須タスク」「任意タスク」「延期タスク」に分割
- 進捗評価は成果物単位（コード、CSV、図、本文）で実施
- 研究締切から逆算した優先順位に固定する
```

## 5. 工程別の呼び出し順（標準運用）

### 5.1 前処理・データ点検

1. Product Manager: 週次タスクを確定
2. Data Engineer: データ整合チェックと処理計画
3. Code Reviewer: 前処理スクリプトの欠陥確認

### 5.2 特徴量作成・統合

1. Data Engineer: 都市構造パラメータ算出フロー確認
2. Code Reviewer: 近傍変数算出と結合処理のレビュー
3. Reality Checker: 再現手順と中間成果物の監査

### 5.3 モデル・評価

1. Product Manager: 実験計画（RQ1-RQ3）を固定
2. Data Engineer: 学習用データセット作成手順を固定
3. Code Reviewer: 学習・評価コードの妥当性チェック
4. Academic Geographer: 空間解釈の妥当性チェック

### 5.4 執筆・提出前

1. Technical Writer: 手法・結果・限界の本文整形
2. Reality Checker: 根拠と再現性の最終監査
3. Product Manager: 残課題の切り分けと提出版確定

## 6. 出力品質チェック表（提出前）

| チェック項目 | 合格基準 |
|--------------|----------|
| LST単位 | すべて℃表記で統一 |
| CRS整合 | 解析手順で座標系の扱いが明記されている |
| スケール比較 | 0m、30-60m、60-90m、90-120mの比較結果がある |
| 再現性 | 入力、前処理、パラメータ、出力先が追跡可能 |
| 根拠提示 | 主要主張に図表または統計根拠が対応 |
| 限界記述 | データ制約と解釈限界が明記されている |
| RQ対応 | RQ1-RQ3それぞれに明確な回答がある |

---

## 7. 運用ルール（軽量版）

1. 1タスクで同時に使うエージェントは最大2体まで。
2. 1回の指示で必ず「対象ファイル」「期待出力」「評価基準」を明示する。
3. 出力を採用する前に、必ず Reality Checker で整合性確認を行う。
4. 実験ログ（入力データ、期間、閾値、乱数シード）を残す。

この最小セットで運用を開始し、必要になった時点でのみ追加エージェントを採用する。

---

## 8. 省力運用プロトコル（指示漏れ防止）

### 8.1 デフォルト自動選択ルール

ユーザーがエージェント指定を省略した場合、以下を自動適用する。

| タスク種別 | 自動適用する観点 |
|------------|------------------|
| Python/GISコードの実装・修正 | Data Engineer + Code Reviewer |
| 前処理・特徴量作成・データ統合 | Data Engineer + Reality Checker |
| モデル結果の解釈（空間差の議論） | Academic Geographer + Reality Checker |
| 論文本文・方法章・結果章の作成 | Technical Writer + Reality Checker |
| 計画・優先順位・週次整理 | Product Manager |

### 8.2 迷った時の固定ルール

1. 実装系は、まず Data Engineer を起点にする。
2. 出力採用前は、必ず Reality Checker で最終確認する。
3. 空間解釈を含む場合は、Academic Geographer を追加する。

### 8.3 短縮指示（コピペ用）

毎回長文を書く代わりに、以下の短文で運用する。

```text
通常モードで進めてください（最小セット自動選択）。
```

```text
厳密モードで進めてください（最小セット自動選択 + 最終的にReality Checker観点を必須適用）。
```

```text
実装モードで進めてください（Data Engineer + Code Reviewer固定）。
```

### 8.4 指示漏れ時の補正方針

ユーザー指示にエージェント指定が無い場合でも、タスク内容から本プロトコルで補完して進める。
ただし、ユーザーが明示的に「通常の単純回答のみ」を指定した場合は適用しない。
