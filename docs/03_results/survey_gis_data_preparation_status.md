## 📋 目的

本ドキュメントは、**測量由来 GIS データの整理・加工・内容確認の進捗**を記録するための結果ドキュメントである。  
LST や公開 GIS を含む研究全体の進捗ではなく、あくまで `Full` シナリオで使う候補である測量由来 GIS の「現在地」を明示することを目的とする。

```python
gdf = gdf.set_crs(epsg=3405, allow_override=True)  # VN-2000手動設定

```

## 3. 測量由来 GIS を研究にどう使うか

### 3.1 現時点で使える可能性が高いもの

- `DC`: 建物被覆率、建物密度
- `GT`: 道路密度
- `RG`: 分析対象域の定義

### 3.2 使い方をまだ詰める必要があるもの

- `DH`: 標高点を 30m グリッドにどう割り当てるか
- `TH`: 水域率の入力としてそのまま使えるか
- `TV`: 植生率や土地利用率へどう落とし込むか

### 3.3 現段階での判断

測量由来 GIS は **「使えるかどうか」ではなく「どう使うかを詰める段階」**にある。  
したがって、現在の課題は前処理そのものではなく、**都市構造パラメータ定義との対応づけ**である。

## 5. データ統合の準備状況

### 5.1 座標系の統一方針

- 測量由来 GIS は WGS84 へ変換済み
- `Full` シナリオでは、LST と同じ WGS84 で位置合わせし、面積・距離計算のみ投影座標系に変換する

### 5.2 空間範囲の整合性

- 測量由来 GIS は ROI 全体より狭い
- `Full` シナリオでは、ROI 全体ではなく **測量 GIS 有効域内の LST** を対象にする必要がある
- この点は `Satellite Only` や `Limited` と異なる

### 5.3 Full シナリオへの接続条件

`Full` を始める前提として、少なくとも次を満たす必要がある。

1. RG / CS を用いた分析対象域の定義
2. DC / GT / TH / TV / DH の変数対応表の確定
3. `calc_urban_params.py` で測量由来 GIS を安定して処理できること

## 7. 制約事項・リスク

### 技術的制約

1. DGNv8 形式は GDAL で直接扱えない
2. 外れ値ジオメトリにより全体 BBox が歪むレイヤがある
3. レイヤ意味は概ね整理できたが、変数定義への変換はまだ確定していない

### データ品質リスク
- **GIS-LST空間範囲の不一致**: GISデータは矩形領域（都市部の一部）、LSTはハノイ全域 → クリップ処理が必須
- **データ欠落（DC/TV）**: 
  - DC: F-48-68-(251-c)（処理ハング、約1.25%欠落）
  - TV: F-48-80-(11-c)（DGNv8形式、約1.25%欠落）
  - 影響: 特定矩形領域（測量図葉2区画分）の都市構造データが不完全
- **測量データの網羅性**: 全7種類のGISデータが都市構造パラメータ算出に十分かの検証が必要

### 研究上のリスク

- `Full` が先に進みすぎると、`Limited` との比較軸が曖昧になる
- 測量由来 GIS は高精度だが、適用範囲が限定的であるため、一般化しすぎる解釈は危険
